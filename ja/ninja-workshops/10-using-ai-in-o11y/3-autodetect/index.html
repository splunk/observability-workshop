AutoDetect と ML 駆動の異常検知

AutoDetect と ML 駆動の異常検知

4 min

AutoDetect とは？ #

AutoDetectは、手動でのしきい値設定を必要とせずに、メトリクスに対してインテリジェントなディテクターを自動的に作成する機械学習を活用した機能です。静的なしきい値を推測する代わりに、AutoDetectはメトリクスの正常な動作パターンを学習し、逸脱が発生したときにアラートを発します。

AutoDetect の仕組み #

AutoDetectは複数のML技術を使用します：

  1. ベースライン学習: 過去のデータを分析して正常なパターンを理解します
  2. 季節性認識: 日次、週次、月次のパターンを認識します
  3. 動的しきい値: メトリクスの変動性に基づいて感度を調整します
  4. コンテキスト異常: よりスマートなアラートのために複数のシグナルを一緒に考慮します

ML 駆動ディテクターの種類 #

急激な変化の検知 #

学習したパターンを超えてメトリクスが急激にスパイクまたは低下したときにアラートを発します。

履歴異常検知 #

時間帯や曜日のパターンを考慮して、現在の値を過去の基準値と比較します。

リソースディテクター #

一般的なインフラストラクチャリソース（CPU、メモリ、ディスク）用の事前設定されたMLディテクターです。

ハンズオン演習：AutoDetect ディテクターの作成 #

演習

ステップ 1：ディテクター作成画面への移動 #

  1. Alerts & Detectors → Detectors に移動します
  2. New Detector をクリックします
  3. AutoDetect または From Template を選択します

ステップ 2：メトリクスの選択 #

  1. 一貫したトラフィックがあるメトリクスを選択します（例：demo.trans.latency または cpu.utilization）
  2. 関連するフィルター（環境、サービスなど）を追加します
  3. チャートを確認してデータが流れていることを確認します

ステップ 3：ML 設定の構成 #

  1. Sudden Change または Historical Anomaly モードを選択します
  2. 感度を調整します：
    • Low: アラート数が少なく、大きな逸脱のみを検知
    • Medium: バランスの取れたアプローチ（推奨）
    • High: より敏感で、微妙な変化も検知
  3. 観測ウィンドウ（考慮する履歴の長さ）を設定します

ステップ 4：アラート設定の構成 #

  1. アラートの重要度（Critical、Warning、Info）を設定します
  2. 通知の受信者を設定します
  3. アラートメッセージをカスタマイズします
  4. 確認してディテクターを有効化します

ML ディテクターの動作を理解する #

学習期間 #

AutoDetectディテクターはベースラインを確立するために時間が必要です：

  • 最小: 24時間のデータ
  • 推奨: 安定したベースラインのために1〜2週間
  • 季節パターン: 週次パターンのために4週間以上

感度のチューニング #

感度設定はディテクターの検知強度を制御します：

text 
Low Sensitivity    → Fewer false positives, might miss subtle issues
Medium Sensitivity → Balanced (default)
High Sensitivity   → Catches more anomalies, more noise possible

ベストプラクティス #

  1. Medium 感度から始める: アラート量に基づいて調整します
  2. 適切なメトリクスを使用する: AutoDetectは以下のようなメトリクスで最も効果的です：
    • 明確なパターンを持つメトリクス（レイテンシ、リクエストレート）
    • 安定した連続的なデータストリーム
    • 十分な履歴データ
  3. 関連するディメンションでグループ化: タグを使用して焦点を絞ったディテクターを作成します
  4. 学習時間を確保: 最初の48時間で効果を判断しないでください
  5. レビューとチューニング: トリガーされたアラートを定期的にレビューして感度を調整します

AutoDetect と静的しきい値の使い分け #

AutoDetect を使用する場合…静的しきい値を使用する場合…
メトリクスに自然な変動がある既知の固定された制限がある
パターンが時間とともに変化する要件が規制や契約に基づいている
トラフィックが季節的または周期的シンプルな二値条件（稼働/停止）
「正しい」しきい値がわからないしきい値が確立されている

AutoDetect のパフォーマンス監視 #

MLディテクターを作成した後：

  1. アラート履歴のレビュー: 誤検知・検知漏れを確認します
  2. 感度の調整: アラートの品質に基づいて微調整します
  3. ベースラインの更新: MLモデルは変化に自動的に適応します
  4. 従来のディテクターとの比較: MLがより早く問題を検知するか確認します

ヒント

AutoDetectは、ユーザートラフィック、トランザクション量、APIリクエストレートなど、時間帯や曜日によって変動するメトリクスに特に効果的です。

よくある落とし穴 #

  • データ不足: パターンを学習するのに十分な履歴が必要です
  • ディメンションが多すぎる: 多くのタグで分割するとMLモデルが希薄化する可能性があります
  • 不安定なメトリクス: 非常に不規則なメトリクスはノイズの多いアラートを生成する可能性があります
  • 最近のデプロイ: 新しいサービスにはベースラインデータがありません
