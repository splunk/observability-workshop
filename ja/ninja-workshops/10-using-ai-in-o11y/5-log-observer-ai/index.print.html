AIを活用したログ分析

Log Observer の AI 機能

Splunk Observability CloudのLog Observerには、大量のログデータを理解するのに役立つAI搭載機能がいくつか含まれています：

  • パターン検出: ログ内の一般的なパターンを自動的に識別
  • 異常のハイライト: 異常なログエントリやパターンを表面化
  • ログクラスタリング: 類似したログメッセージをグループ化
  • インテリジェントフィルタリング: コンテキストに基づいたAI推奨フィルター
  • 自然言語クエリ: 自然言語を使用したログのクエリ（利用可能な場合）

パターン検出

ログパターン検出とは

個々のログ行を表示する代わりに、パターン検出はログを構造でグループ化し、以下のことを容易にします：

  • 頻出パターンと稀なパターンの識別
  • 新しいまたは異常なログパターンの発見
  • 異常なエントリへの集中
  • 繰り返しログからのノイズ削減

仕組み

AIエンジンは：

  1. ログメッセージの構造を分析
  2. 可変部分（ID、タイムスタンプ、値）を識別
  3. パターンテンプレートを作成
  4. テンプレートでログをグループ化
  5. パターンの頻度と変化を追跡

ハンズオン演習：ログパターンの探索

演習

ステップ 1: Log Observer にアクセス

  1. Log Observer に移動します
  2. アクティブなログデータがある時間範囲を選択します
  3. 多様なログを持つサービスまたはインデックスを選択します

ステップ 2: パターンビューを有効化

  1. Patterns または Pattern Analysis ビューを探します
  2. 個別ログビューからパターンビューに切り替えます
  3. ログがどのようにクラスタリングされているかを観察します

ステップ 3: パターンを分析

パターンリストを確認します：

  • 高頻度パターン: 通常の、予想されるログ
  • 新しいパターン: 最近出現した（潜在的な問題）
  • 稀なパターン: 低頻度、調査の価値あり
  • エラーパターン: 構造化されたエラーメッセージ

ステップ 4: 異常を調査

  1. 稀なパターンまたは新しいパターンをクリックします
  2. そのパターンのログ例を表示します
  3. ベースライン/通常パターンと比較します
  4. 問題を示しているかどうかを判断します

ログ異常検出

検出される異常の種類

  1. 頻度の異常: ログ量の急激な増加/減少
  2. コンテンツの異常: 異常なフィールド値またはメッセージ内容
  3. パターンの異常: ベースラインで見られない新しいパターン
  4. タイミングの異常: 異常な時間に出現するログ

異常検出の設定

MLを使用したログベースのディテクターを設定します：

  1. Alerts & Detectors に移動します
  2. Log Observerから新しいディテクターを作成します
  3. Anomaly Detection モードを選択します
  4. 設定項目：
    • ベースライン期間
    • 感度レベル
    • アラート条件
    • 通知チャンネル

インテリジェントログフィルタリング

AI 推奨フィルター

調査中に、Log Observerは以下を提案する場合があります：

  • 関連フィールド: フィルタリングする価値のあるタグや属性
  • 一般的な値: 頻繁に出現するフィールド値
  • 異常な値: 注目に値する異常なフィールド値
  • 相関属性: 一緒に出現することが多いフィールド

推奨フィルターの使用

  1. UIでフィルターの提案を探します
  2. クリックして推奨フィルターを適用します
  3. 結果に基づいて絞り込みます
  4. 有用なフィルターの組み合わせを保存します

APM およびインフラストラクチャとのログ相関

自動コンテキストリンク

AIはログを以下と接続するのを助けます：

  • トレース: ログエントリを分散トレースにリンク
  • サービス: ログをAPMサービスに関連付け
  • インフラストラクチャ: ホスト、コンテナ、Podに接続
  • メトリクス: ログパターンをメトリクスの変化と相関

AI のパンくずリストをたどる

調査中：

  1. ログエントリまたはパターンから開始します
  2. Related Content の提案を探します
  3. 相関するトレース、メトリクス、またはインフラストラクチャにジャンプします
  4. Tag Spotlightを使用して問題を絞り込みます
  5. ログに戻って調査結果を確認します

ログの要約とインサイト

主要インサイトパネル

AI生成のインサイトには以下が含まれる場合があります：

  • エラー率のサマリー: エラータイプ別にグループ化
  • サービスの健全性: ログの重大度に基づく
  • トレンド分析: 時間の経過に伴うログパターンの変化
  • 比較分析: 現在とベースライン期間の比較

インサイトの例

⚠️ Error rate increased 340% in the last hour
   → Top error: "Database connection timeout" (1,247 occurrences)
   → Affected services: checkout-service, payment-service
   → Started at: 14:23 UTC

📊 New log pattern detected
   → "WARN: Cache miss for key {key}" appeared 892 times
   → First seen: 14:25 UTC
   → May indicate cache invalidation issue

AI を活用したログ分析のベストプラクティス

1. ログを構造化する

構造化ログを使用してAIを支援します：

{
  "timestamp": "2024-01-15T14:23:45Z",
  "level": "ERROR",
  "service": "checkout-service",
  "message": "Payment processing failed",
  "error_code": "PAYMENT_TIMEOUT",
  "transaction_id": "txn_123456",
  "customer_tier": "enterprise"
}

2. 一貫したフィールド名を使用する

  • サービス間でフィールド命名を標準化します
  • 共通のタクソノミーを使用します（例：serviceName や service_id ではなく service.name）
  • すべてのログに必須のコンテキストを含めます

3. 適切なログレベルを設定する

  • DEBUG: 詳細な診断情報（開発用）
  • INFO: 一般的な情報メッセージ
  • WARN: 潜在的に有害な状況
  • ERROR: アプリが継続できる可能性のあるエラーイベント
  • FATAL: 早期終了を引き起こす重大なエラー

4. ログサンプリングを活用する

大量のログの場合：

  • AIを使用して代表的なサンプルを識別します
  • エラーログと異常に焦点を当てます
  • インテリジェントサンプリングを適用してコストを削減します

5. ログベースのディテクターを作成する

以下に対するアラートを設定します：

  • 重大なエラーパターン
  • 異常なログ量
  • 新しいエラータイプ
  • セキュリティ関連のパターン

ユースケース

ユースケース 1: メモリリークの特定

観察結果: パターン分析で「GC pressure」警告の増加を表示

AI の支援：

  • GC関連のログパターンをグループ化
  • 頻度の増加をハイライト
  • メモリメトリクスと相関
  • 影響を受けるサービスのトレースにリンク

ユースケース 2: セキュリティ問題の検出

観察結果: 新しいパターン「Authentication failed」が出現

AI の支援：

  • 新規/稀なパターンとしてフラグ付け
  • ソースIPでクラスタリング
  • 異常なアクセスパターンをハイライト
  • セキュリティ関連のフィルターを提案

ユースケース 3: データベースパフォーマンスの低下

観察結果: スロークエリの警告が増加

AI の支援：

  • パターンでクエリをグループ化
  • 最も遅いクエリタイプを特定
  • データベースメトリクスと相関
  • アプリケーショントレースにリンク

制限事項と考慮事項

  • パターンの品質はログ構造に依存: 非構造化ログはパターン化が困難
  • 高カーディナリティフィールド: UUIDや一意のIDはパターンを分割する可能性
  • 学習期間: AIは異常検出のためにベースラインデータが必要
  • コンテキストが重要: ログAIを他のオブザーバビリティシグナルと組み合わせる
ヒント

最も効果的なログ分析は、AIを活用したパターン検出とドメイン知識を組み合わせます。AIを使用してシグナルを表面化し、あなたの専門知識を適用して解釈します。
