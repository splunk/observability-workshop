単一サービスのエラーログに絞り込みました。次に、問題が*いつ*始まったのか、エラーが*どのような*ものかを理解しましょう。

{{% notice title="Exercise" style="green" icon="running" %}}

**ステップ 1: タイムラインを分析する**

* ページ上部のタイムラインチャートを確認します。エラーバーの形状に注目してください。

{{< tabs >}}
{{% tab title="Question" %}}
**エラーレートは一定ですか、それとも特定の時間にスパイクしましたか？それは何を示唆していますか？**
{{% /tab %}}
{{% tab title="Answer" %}}
突然のスパイクは、その時点でのデプロイメント、設定変更、または外部依存関係の障害を示唆します。一定のエラーレートは、サービスの起動以来問題が存在していたことを示唆します。どちらのパターンも根本原因の手がかりとなります。
{{% /tab %}}
{{< /tabs >}}

**ステップ 2: ログエントリを展開する**

* テーブル内の代表的なエラーログエントリをクリックして展開します。
* 詳細ペインには、ログエントリに付加されたすべての構造化フィールドが表示されます。

以下の主要フィールドを確認します

| フィールド | 情報の内容 |
|-------|-------------------|
| `message` | 実際のエラーテキスト — 根本原因が含まれていることが多いです |
| `service.name` | ログを出力したサービスを確認します |
| `k8s.pod.name` | 特定の Pod インスタンス |
| `deployment.environment` | ワークショップ環境 |
| `severity` | ログレベル（ERROR であるはずです） |

**ステップ 3: パターンを探す**

* 複数のエラーエントリをスクロールして確認します。すべて同じメッセージですか、それとも異なるエラータイプがありますか？
* ログメッセージ内のバージョンタグ、デプロイメント識別子、または依存関係の参照に注目してください。

{{< tabs >}}
{{% tab title="Question" %}}
**すべてのエラーログが同じメッセージパターンを共有していますか、それとも複数の異なるエラーがありますか？**
{{% /tab %}}
{{% tab title="Answer" %}}
<!-- TODO: Update with actual patterns from OTel Demo v2.0.1 -->
繰り返されるメッセージテンプレートを探してください。すべてのエラーが同じパターンを共有している場合、根本原因は1つである可能性が高いです。複数のパターンがある場合は、カスケード障害を示している可能性があります。
{{% /tab %}}
{{< /tabs >}}

{{% /notice %}}

{{% notice title="Info" style="info" %}}
OpenTelemetry による構造化ログは、すべてのログエントリが一貫したクエリ可能なフィールドを持つことを意味します。これが Log Observer のノーコードフィルタリングを効果的にしている理由です — 構造はデータに組み込まれており、事後に付与されたものではありません。
{{% /notice %}}
