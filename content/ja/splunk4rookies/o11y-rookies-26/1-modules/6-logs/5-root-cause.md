適切なサービスにフィルタリングし、タイムラインを分析し、エラーパターンを確認しました。次は根本原因を特定します。

{{% notice title="Exercise" style="green" icon="running" %}}

* ログテーブルのエラーエントリをクリックして、詳細ペインを開きます。
* `message` フィールドを注意深く読みます。

{{< tabs >}}
{{% tab title="Question" %}}
**エラーメッセージに基づいて、根本原因は何ですか？開発チームに何を推奨しますか？**
{{% /tab %}}
{{% tab title="Answer" %}}
<!-- TODO: Update with actual root cause from OTel Demo v2.0.1 fault injection -->
エラーメッセージは具体的な障害を示しています。これに基づいて、開発チームに障害が発生しているコンポーネントの調査を推奨します — 無効な設定、期限切れの認証情報、不足している依存関係、またはコードの欠陥のいずれかです。
{{% /tab %}}
{{< /tabs >}}

<!-- TODO screenshot: Log entry detail showing the root cause error message -->

* **X** をクリックしてログエントリの詳細ペインを閉じます。

{{% /notice %}}

{{% notice style="blue" title="Congratulations" icon="wine-bottle" %}}

スタンドアロンの調査ツールとして **Splunk Log Observer** を正常に使用できました。事前のコンテキストなし — トレースもメトリクスダッシュボードもない状態からゼロから始めて、以下を実行しました

1. Log Observer を開き、環境フィルターを設定しました
2. 重大度でグループ化し、エラーでフィルタリングしました
3. 障害が発生しているサービスを特定しました
4. エラーのタイムラインを分析し、問題がいつ始まったかを把握しました
5. ログエントリを検査して根本原因を見つけました

すべて Log Observer 内で、ポイント＆クリックのフィルタリングのみを使用して実行しました。

{{% /notice %}}

## まとめ

| テクニック | 表示された内容 |
|-----------|---------------|
| 重大度でグループ化 | すべてのサービスにわたるエラー vs 情報 vs デバッグの分布 |
| サービスでフィルタリング | ノイズから障害コンポーネントを分離 |
| タイムライン分析 | 問題がいつ始まったか — スパイク vs 定常 |
| ログエントリの検査 | 具体的なエラーメッセージと根本原因 |

{{% notice title="Info" style="info" %}}
このシナリオでは、Log Observer が**主要な**調査ツールでした。実際には、APM トレースから **Related Content** を経由して Log Observer に到達することもできます — これは `trace_id` などの共有識別子を使用してトレースとログを自動的に関連付ける機能です。どちらのパスも有効です。適切な開始点は、インシデント発生時に利用可能なシグナルに依存します。
{{% /notice %}}
