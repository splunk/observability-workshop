適切なサービスへの絞り込み、タイムラインの分析、エラーパターンの調査を行いました。次は根本原因を特定する番です。

{{% notice title="エクササイズ" style="green" icon="running" %}}

* ログテーブル内のエラーエントリをクリックして、詳細ペインを開きます。
* `message` フィールドを注意深く読みます。

{{< tabs >}}
{{% tab title="質問" %}}
**エラーメッセージに基づくと、根本原因は何ですか？開発チームには何を提案しますか？**
{{% /tab %}}
{{% tab title="回答" %}}
<!-- TODO: Update with actual root cause from OTel Demo v2.0.1 fault injection -->
エラーメッセージは具体的な障害を明らかにしています。これに基づき、開発チームには問題のあるコンポーネントを調査するよう提案することになります。原因が無効な設定、期限切れの認証情報、欠落した依存関係、コード欠陥のいずれであるかを確認します。
{{% /tab %}}
{{< /tabs >}}

* **X** をクリックしてログエントリの詳細ペインを閉じます。

{{% /notice %}}

{{% notice style="blue" title="お疲れ様でした" icon="wine-bottle" %}}

**Splunk Log Observer** をスタンドアロンの調査ツールとして使いこなせるようになりました。事前のコンテキスト（トレースもメトリクスダッシュボードも）が一切ない状態から始めて、以下を実施しました。

1. Log Observer を開いて環境フィルターを設定する
2. 重大度でグループ化し、エラーに絞り込む
3. 障害の発生しているサービスを特定する
4. エラータイムラインを分析して問題が発生した時刻を把握する
5. ログエントリを調査して根本原因を見つける

すべて Log Observer 内で、ポイントアンドクリックのフィルタリングのみで完了しました。

{{% /notice %}}

## まとめ

| 手法 | 確認できたこと |
|-----------|---------------|
| 重大度でグループ化 | 全サービスにわたる error / info / debug の分布 |
| サービスでフィルタリング | ノイズの中から障害の発生しているコンポーネントを切り分け |
| タイムライン分析 | 問題が発生した時刻 — スパイクか継続的か |
| ログエントリ調査 | 具体的なエラーメッセージと根本原因 |

{{% notice title="情報" style="info" %}}
このシナリオでは、Log Observer が**主要な**調査ツールでした。実際の運用では、APM のトレースから **Related Content** 経由で Log Observer にたどり着くこともできます。これは `trace_id` などの共通識別子を使ってトレースとログを自動的に関連付ける機能です。どちらの経路も有効であり、最適な出発点はインシデント発生時点で利用可能なシグナルによって異なります。
{{% /notice %}}
