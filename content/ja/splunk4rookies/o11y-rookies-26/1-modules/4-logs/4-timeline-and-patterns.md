ここまでで、単一サービスからのエラーログにまで絞り込めました。次は、問題が*いつ*始まったのか、そしてエラーが*どのような*内容なのかを理解しましょう。

{{% notice title="Exercise" style="green" icon="running" %}}

**Step 1: タイムラインを分析する**

* ページ上部のタイムラインチャートを確認します。エラーバーの形に注目してください。

{{< tabs >}}
{{% tab title="Question" %}}
**エラーレートは一定ですか、それとも特定の時刻にスパイクしていますか？そこから何が読み取れるでしょうか？**
{{% /tab %}}
{{% tab title="Answer" %}}
急なスパイクは、その時点でデプロイ、設定変更、または外部依存先の障害が発生したことを示唆しています。一定のエラーレートは、サービス起動以降ずっと問題が存在していることを示唆します。どちらのパターンも、根本原因を探るための手がかりとなります。
{{% /tab %}}
{{< /tabs >}}

**Step 2: ログエントリを展開する**

* テーブルの中から代表的なエラーログエントリをクリックして展開します。
* 詳細パネルには、そのログエントリに付与されているすべての構造化フィールドが表示されます。

以下の主要フィールドを確認してください。

| フィールド | 何を示すか |
|-------|-------------------|
| `message` | 実際のエラーテキスト — 多くの場合、根本原因を含みます |
| `service.name` | どのサービスがログを発行したかを確認できます |
| `k8s.pod.name` | 特定のPodインスタンス |
| `deployment.environment` | お使いのワークショップ環境 |
| `severity` | ログレベル（ERRORになっているはずです） |

**Step 3: パターンを探す**

* いくつかのエラーエントリをスクロールして確認します。すべて同じメッセージですか、それとも異なる種類のエラーがありますか？
* ログメッセージ内のバージョンタグ、デプロイ識別子、依存先への参照などに注目してください。

{{< tabs >}}
{{% tab title="Question" %}}
**すべてのエラーログは同じメッセージパターンを持っていますか、それとも複数の異なるエラーが存在しますか？**
{{% /tab %}}
{{% tab title="Answer" %}}
<!-- TODO: Update with actual patterns from OTel Demo v2.0.1 -->
繰り返し現れるメッセージテンプレートを探してください。すべてのエラーが同じパターンを共有していれば、おそらく根本原因はひとつです。複数のパターンが見られる場合は、連鎖的な障害が発生している可能性があります。
{{% /tab %}}
{{< /tabs >}}

{{% /notice %}}

{{% notice title="Info" style="info" %}}
OpenTelemetryによる構造化ロギングでは、すべてのログエントリが一貫したクエリ可能なフィールドを持ちます。これこそが、Log Observerのノーコードフィルタリングを効果的にしている理由です — 構造はデータに後付けされたものではなく、最初から組み込まれているのです。
{{% /notice %}}
