このワークショップでは、Observability ワークフロー内にアプリケーションセキュリティを組み込むメリットについて紹介します。これらのモジュールを完了すると、エンジニアリングチームとセキュリティチームがランタイムのリスクビューを共有し、CVSS だけに頼らない優先順位付けを行い、既存の SOC ツールに結果をフィードする方法を理解できます。

このワークショップは、アプリケーションオーナー、SRE、および SecOps チームに適しています。

---

## 概要

アプリケーションセキュリティのデータは、スタンドアロンのスキャナー、スプレッドシート、本番環境で実際に稼働しているものを反映しないチケットキューに散在していることがよくあります。その結果、修復キューがエンジニアリングとセキュリティの間のギャップを広げ、悪用可能な問題に対する MTTR が遅くなります。これは通常、以下の3つの理由で発生します

- **可視性の断片化**: セキュリティと信頼性が、共有サービスコンテキストのない別々のツールで評価されています。
- **理論的な優先順位付け**: チームは、観測された悪意のあるアクティビティや、デプロイされたライブラリやサービスに紐づくランタイム攻撃シグナルなしに、CVSS だけでリスクを評価しています。
- **運用・ワークフロー負債**: 手動トリアージ、重複する SecOps ワークフロー、管理されていないライブラリの乱立が修復能力を消費しています。

これらの課題に対処するために、以下の方法が必要です

- **可視性の維持**: 2つ目のエージェントや製品を追加することなく、既存の共有ワークスペースで信頼性とセキュリティを統合します。
- **コンテキストに基づく更新とアップグレード**: エンジニアリングに依頼する前に、脆弱性のリスクプロファイル、ライブラリ名、バージョン、および影響を受けるサービスの影響範囲に修復ガイダンスを紐づけます。
- **技術的負債の排除**: 共有コンテキスト全体でステータスライフサイクル管理とハイジーンにより脆弱性キューを管理します。
- **既知の脅威の優先順位付け**: CVSS スコアと悪用リスクを比較し、カタログ化された CVE からコードレベルのスタックトレースを含むランタイム攻撃フォレンジクスへピボットします。
- **防御の近代化**: 定期的なスキャンとリアクティブな緊急対応から、Splunk Security と統合された継続的なランタイム検出に移行します。
