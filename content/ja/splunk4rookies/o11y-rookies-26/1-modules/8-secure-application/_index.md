このワークショップでは、Observability ワークフロー内にアプリケーションセキュリティを組み込むメリットを紹介します。これらのモジュールを終了するまでに、エンジニアリングチームとセキュリティチームがリスクのランタイムビューを共有し、CVSS だけにとらわれない優先順位付けを行い、既存の SOC ツールに結果をフィードする方法を理解できるようになります。

このワークショップは、アプリケーションオーナー、SRE、および SecOps チームに適しています。

---

## 概要

アプリケーションセキュリティデータは、スタンドアロンスキャナー、スプレッドシート、本番環境で実際に稼働しているものを反映しないチケットキューに分散していることが多いです。その結果、修復キューがエンジニアリングとセキュリティの間のギャップを広げ、悪用可能な問題に対する MTTR が遅くなります。これは通常、3 つの理由で発生します

- **可視性の断片化**: セキュリティと信頼性が、共有サービスコンテキストのない別々のツールで評価されています。
- **理論的な優先順位付け**: チームが観測された悪意のあるアクティビティや、デプロイされたライブラリやサービスに紐づくランタイム攻撃シグナルなしに、CVSS だけでリスクを評価しています。
- **運用およびワークフローの負債**: 手動トリアージ、重複する SecOps ワークフロー、管理されていないライブラリの肥大化が修復能力を消費しています。

これらの課題に対処するには、以下を実現する方法が必要です

- **可視性の維持**: 2 つ目のエージェントや製品を追加することなく、既存の共有ワークスペースで信頼性とセキュリティを統合します。
- **コンテキストを伴う更新とアップグレード**: エンジニアリングに依頼する前に、修復ガイダンスを脆弱性リスクプロファイル、ライブラリ名、バージョン、および影響を受けるサービスの影響範囲に紐づけます。
- **技術的負債の排除**: 共有コンテキスト全体でステータスライフサイクル管理と衛生管理により、脆弱性キューを統制します。
- **既知の脅威の優先順位付け**: CVSS スコアと悪用リスクを比較し、カタログ化された CVE からコードレベルのスタックトレースを伴うランタイム攻撃フォレンジクスにピボットします。
- **防御の近代化**: 定期的なスキャンと事後対応的な緊急対応から、Splunk Security と統合された継続的なランタイム検出に移行します。
