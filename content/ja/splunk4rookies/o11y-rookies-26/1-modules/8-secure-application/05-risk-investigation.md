## 調査にランタイムコンテキストが必要な理由

影響範囲の分析なしにパッチを適用すると、悪用の機会が残ります。エンジニアリングチームが1つのサービスにのみ影響するライブラリをアップグレードする一方で、同じ CVE が他の重要なパスに広がっている場合や、チームが孤立した発見を過度にエスカレーションしてしまう場合があります。

Splunk Secure Application は、**ライブラリ名とバージョン**、**修正ガイダンス**、および**影響を受けるサービスの列挙**を1つの詳細ビューに統合し、アップグレードの決定が実際にデプロイされている内容に基づいて行われるようにします。

---

## 5.1 ガイド付き脆弱性トラブルシューティング

1. **`ad`** サービスの Application Security ビューから、前のモジュールで特定した**高 CVSS / 高リスクスコア**の脆弱性をクリックします。
2. 脆弱性の詳細パネルを確認します

| フィールド | 確認すべき内容 |
|-------|------------------|
| **Impacted libraries** | パッケージ名と固定バージョン |
| **External references** | アドバイザリおよびベンダーガイダンスへのリンク |
| **Description** | 脆弱性に関する包括的な詳細 |
| **Recommended actions** | 回避策と修正オプション |

![apm](./images/05-runtime-details.png)

> *「複数のポータルにわたる面倒な調査は不要です - コンテキストは同じ場所にあります。」*

---

## 5.2 影響を受けるサービスの評価（影響範囲）

1. 脆弱性の詳細の下部にある **Affected services** セクションまでスクロールします。
2. 同じ CVE が他の重要なサービスにも及んでいるのか、`ad` に限定されているのかを判断します。

![apm](./images/05-runtime-services.png)

> [!NOTE]
> 影響を受けるサービスに移動して、そのサービスに関連する他の脆弱なライブラリを確認することもできます。調査中の単一の CVE を超えて、注意が必要な他のデプロイ済みパッケージを確認できます。

> *「アプリケーションチームに連絡する前に、利用可能なすべてのリソースと外部参照を確認して、スタック全体にわたる拡張リスクエクスポージャーを把握できます。」*

---

## 学んだこと

- ライブラリバージョンのコンテキストを含む統合された修正ガイダンスへのアクセス方法。
- 影響を受けるサービスの列挙がアップグレードの優先順位付けにどのように役立つか。
- サービススコープのライブラリインベントリがエンジニアリングへの引き継ぎをどのように強化するか。
