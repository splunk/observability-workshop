## コンポーネントレビュー

Jenkins からメトリクスを取得するために必要なコンポーネントの種類を振り返ります

{{% tabs %}}
{{% tab title="Extension" %}}
Extension が解決するビジネスユースケースは以下の通りです

1. ランタイム設定を必要とする共有機能を持つこと
1. Collector のランタイムの観測を間接的に支援すること

詳細は [**Extensions の概要**](../2-extensions) を参照してください。
{{% /tab %}}
{{% tab title="Receiver" %}}
Receiver が解決するビジネスユースケースは以下の通りです

- リモートソースからデータをフェッチする
- リモートソースからデータを受信する

これは一般的に _pull_ 型 vs _push_ 型のデータ収集と呼ばれます。詳細は [Receiver の概要](../3-receivers) で確認できます。
{{% /tab %}}
{{% tab title="Processor" %}}
Processor が解決するビジネスユースケースは以下の通りです

- データ、フィールド、または値の追加や削除
- データの観測と判断
- バッファリング、キューイング、および並べ替え

注意すべき点は、Processor を流れるデータ型は、下流のコンポーネントに同じデータ型を転送する必要があるということです。詳細は [Processor の概要](../4-processors) を参照してください。

{{% /tab %}}
{{% tab title="Exporter" %}}
Exporter が解決するビジネスユースケースは以下の通りです

- データをツール、サービス、またはストレージに送信する

OpenTelemetry Collector は「バックエンド」、つまりオールインワンのオブザーバビリティスイートになることを意図していません。むしろ、OpenTelemetry の設立理念である「すべての人のためのベンダーに依存しないオブザーバビリティ」の原則を守り続けます。詳細を振り返るには、[**Exporter の概要**](../5-exporters) を参照してください。

{{% /tab %}}
{{% tab title="Connectors" %}}

これはワークショップでは取り上げなかったコンポーネントタイプです。Collector に比較的新しく追加されたものですが、Connector を理解する最良の方法は、異なるテレメトリタイプやパイプライン間で使用できる Processor のようなものと考えることです。つまり、Connector はログとしてデータを受け入れてメトリクスを出力したり、あるパイプラインからメトリクスを受け入れて観測したデータに関するメトリクスを提供したりできます。

Connector が解決するビジネスユースケースは以下の通りです

- 異なるテレメトリタイプ間の変換
  - ログからメトリクスへ
  - トレースからメトリクスへ
  - メトリクスからログへ
- 受信データの観測と独自データの生成
  - メトリクスを受け入れ、データの分析メトリクスを生成する

[Processor の概要](../4-processors) の **Ninja** セクション内に簡単な概要がありました。新しい Connector コンポーネントのアップデートについてはプロジェクトを確認してください。
{{% /tab %}}
{{% /tabs %}}

コンポーネントの概要から、Jenkins 用のプルベースの Receiver を開発する必要があることは明らかです。
