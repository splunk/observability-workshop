## カスタムコンポーネントの開発

OpenTelemetry Collectorのコンポーネントを構築するには、3つの重要な要素が必要です。

1. Configuration - _ユーザーに公開される設定値_
1. Factory - _提供された値を使用してコンポーネントを作成する_
1. Business Logic - _コンポーネントが実行すべき処理_

ここでは、プロジェクトの重要なDevOpsメトリクスを追跡できるように、Jenkinsと連携するコンポーネントを構築する例を使用します。

測定対象のメトリクスは以下の通りです。

1. Lead time for changes - _「コミットが本番環境に反映されるまでの時間」_
1. Change failure rate   - _「本番環境で障害を引き起こすデプロイの割合」_
1. Deployment frequency  - _「チームが本番環境へ正常にリリースする頻度」_
1. Mean time to recover  - _「チームが本番環境の障害から復旧するまでの時間」_

これらの指標は、GoogleのDevOps Research and Assessment（DORA）[^1]チームによって、ソフトウェア開発チームのパフォーマンスを示すために特定されました。_Jenkins CI_ を選択した理由は、同じオープンソースソフトウェアのエコシステムに留まることで、将来的にベンダー管理のCIツールが採用するための例として機能できるためです。

## 計装 vs コンポーネント

組織内のオブザーバビリティのレベルを向上させる際には、いくつかのトレードオフがあることを考慮する必要があります。

| | メリット | デメリット |
| ----- | ----- | ----- |
| **（自動）計装** | システムを観測するために外部APIを監視する必要がない。 | 計装の変更にはプロジェクトへの変更が必要。 |
| | システムオーナーや開発者がオブザーバビリティを変更できる。 | 追加のランタイム依存関係が必要。 |
| | システムコンテキストを理解し、キャプチャしたデータを _Exemplars_ と関連付けられる。 | システムのパフォーマンスに影響を与える可能性がある。 |
| **コンポーネント** | データ名やセマンティクスの変更をシステムのリリースサイクルとは独立して展開できる。 | 破壊的なAPI変更にはシステムとCollector間の調整されたリリースが必要。 |
| | 収集データの更新や拡張がユーザーにとってシームレスな変更となる。 | キャプチャされたデータのセマンティクスが、新しいシステムリリースと整合しない形で予期せず壊れる可能性がある。 |
| | サポートチームがオブザーバビリティの実践について深い理解を持つ必要がない。 | システムから表面化できるのは、厳密に外部に公開された情報のみ。 |
