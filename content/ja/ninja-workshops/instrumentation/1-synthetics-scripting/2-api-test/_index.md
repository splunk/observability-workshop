**API Test** は、APIエンドポイントの機能とパフォーマンスを柔軟にチェックする方法を提供します。APIファーストの開発へのシフトにより、コアとなるフロントエンド機能を提供するバックエンドサービスのモニタリングの必要性が高まっています。Real Browser Testでリグレッションが表面化する頃には、基盤となるAPIはすでにしばらく壊れていることが多いためです。

マルチステップのAPIインタラクションのテストに興味がある場合でも、個々のエンドポイントのパフォーマンスを可視化したい場合でも、API Testは目標の達成を支援します。

## API TestとReal Browser Testの違い

API Testは **ヘッドレス** です。ブラウザもJavaScriptエンジンもDOMもありません。テストランナーは生のHTTPリクエストを発行し、レスポンスを解析し、バリデーションルールを実行します。これには3つの実用的な影響があります。

- **実行あたりのコストが低く、高速。** 一般的なAPIの実行は、RBTが要する10〜30秒ではなく、数十〜数百ミリ秒で完了します。これにより高頻度のモニタリングが実現可能になり、多くのチームが重要なAPI Testを毎分実行しています。
- **レンダリングシグナルなし。** 何もレンダリングされないため、描画時間やレイアウトシフトを測定できません。実際のユーザーが結果をどのように *体験する* かを知る必要がある場合は、そのジャーニーをカバーするRBTが依然として必要です。
- **ステップの連鎖が容易。** あるレスポンスから抽出した変数を、次のリクエストで名前で参照できます。これにより「ログイン、ユーザープロファイルの取得、アクションの実行、結果の検証」のような実際のビジネストランザクションを自然にモデル化できます。

## API Testを使用するタイミング

- **バックエンドSLO。** サービスが依存するAPIのエンドポイント可用性とレイテンシ。
- **認証フロー。** OAuthトークンの発行、SAMLアサーション交換、セッション作成 — すべてマルチステップで、すべて重要で、ブラウザテストからの推測よりも明示的にバリデーションする方が容易です。
- **JSONコントラクトバリデーション。** レスポンスに特定のフィールドが含まれること、配列が期待される要素数を持つこと、値が正規表現に一致することをアサートします。クライアントに影響する前に、破壊的なスキーマ変更を検出します。
- **合成ビジネストランザクション。** 実際のユーザーアクションを表すAPI呼び出しのシーケンスを、人間が現在実行していない場合でもエンドツーエンドで実行します。

この章では、公開されているSpotify Web APIに対して2ステップのAPI Testを構築します。最初のステップは **OAuth 2 Client Credentials** 認証を実行し、レスポンスからBearerトークンを抽出します。2番目のステップはそのトークンを使用してSpotify検索エンドポイントを呼び出し、最初に一致したトラックのIDを抽出します。認証の後に認証済みアクションを実行するという同じパターンは、実際のバックエンドモニタリングのユースケースの大部分をカバーします。
