前のステップのScatterplotで、ドットの1つをクリックしてテスト実行データにドリルダウンします。できるだけ最新のテスト実行（最も右側）を選択して、現在のサイト状態に近い実行結果を確認してください。

![Drilldown](../../img/drilldown.png)

この1つのページには、Splunk Synthetic Monitoringがその1つのロケーションからの1回の実行で記録した **すべて** がキャプチャされています。各セクションが何のためにあるかを理解しておく価値があります。これらはリグレッションのトリアージ時に最も多く使うビューです。

## 結果ヘッダー

ページの上部に、実行の概要結果が表示されます。

- **Status** — SuccessまたはFailureが表示され、失敗したステップがあればその名前が示されます（前の章でステップ名を変更したのはこのためです）。
- **Location** — 実行元のAWSリージョン。リージョン固有のリグレッションを追跡している場合はScatterplotと照合します。
- **Timestamp** — 実行が開始された時刻（ローカルタイムゾーン）。
- **Total duration** — ジャーニー全体にかかった時間。

## フィルムストリップとビデオ

上部に並んだスクリーンショットは **フィルムストリップ** です。Synthetic Monitoringは実行中に一定間隔でスクリーンショットをキャプチャするため、ジャーニー全体を視覚的にスクラブできます。右側にはテストが見たページの **ビデオリプレイ** があり、標準プレーヤーでフレームごとにスクラブできます。

フィルムストリップとビデオは「ユーザーが実際に何を見たか？」に答える最も速い方法です。予期しないモーダル、コンテンツシフトエラー、要素の欠落などでステップが失敗した場合に特に有用です。バグを再現可能にするために開発者と共有する必要があるのは、多くの場合これだけです。

## Business transactionsとPages

フィルムストリップの下に2つのサマリー行が表示されます。

- **Business Transactions** — ステップエディターで定義したSyntheticトランザクション（このワークショップでは定義していないため、デフォルトの「Synthetic transaction 1」エントリが表示されます）。各トランザクションは独自のバーとして表示され、それぞれの所要時間が示されます。
- **Pages** — ジャーニーが訪問したすべてのURL（順番に表示）。Online Boutiqueテストでは、ホームページ、Vintage Camera Lensの商品ページ、カート、チェックアウト確認の4つが表示されます。いずれかをクリックすると、下のウォーターフォールがそのページのリソースのみにフィルタリングされます。

## ウォーターフォール

ウォーターフォールは結果ページの主力です。ページが行ったすべてのHTTPリクエストが順番に表示され、それぞれのタイミングとステータスが示されます。フィルターバーでリソースタイプ（**All / XHR / JS / CSS / Img / Media / Font / Doc / WS / Manifest**）を絞り込めます。各行には以下が表示されます。

- **METHOD** — 通常は `GET` ですが、APIへのPOSTもここに表示されます。
- **FILE** — リソースパス。行をクリックするとタイミングの詳細（DNS、connect、TLS、request、response、content download）が展開されます。
- **DOMAIN** — サードパーティの影響を特定するのに有用です。`cdn.googletagmanager.com` やA/BテストSDKが一貫して遅いリストの上位に表示される場合、遅延または削除の候補が見つかったことになります。
- **SIZE** — ネットワーク上のサイズ。ここで予想外のもの（4MBのヒーロー画像、ミニファイされていないバンドルなど）に注意します。
- **STATUS** — HTTPステータスコード。赤いものは調査する価値がありますが、失敗せずにレイテンシを追加する予期しない `301`/`302` チェーンにも注意します。

## Web Vitals

ページの右側に、Splunkはこの実行の [Core Web Vitals](https://web.dev/articles/vitals) を報告します。

- **Largest Contentful Paint (LCP)** — 最大の可視コンテンツ要素（通常はヒーロー画像またはメイン見出し）のレンダリングが完了するまでの時間。良好: ≤ 2.5秒、改善が必要: ≤ 4秒、不良: > 4秒。
- **Total Blocking Time (TBT)** — ページロード中にメインスレッドがロングタスクによってブロックされた合計時間。First Input Delayのラボ環境版です。良好: ≤ 200ms、不良: > 600ms。
- **Cumulative Layout Shift (CLS)** — ロード中にレイアウトが予期せずシフトした量（例: 読み始めた後にバナーが記事を押し下げる）。良好: ≤ 0.1、不良: > 0.25。

これらはGoogleの標準的なWeb Vitalsメトリクスであるため、他のRUMツールと同じ方法でチャート化やアラート設定ができます。RUMとは異なり、Syntheticバージョンは制御された環境から取得されるため、Synthetic LCPのリグレッションはほぼ常にコード変更であり、ユーザー環境の変化ではありません。

## 次のステップ

このビューに慣れたら、本番環境での有用なフォローアップをいくつか紹介します。

- ここで表示されるメトリクス（実行時間、ステップ時間、個々のWeb Vitals）に **Detectorを作成** します。Splunk Synthetic MonitoringのDetectorは他のObservability Cloud Detectorと並んで配置され、同じ通知チャネルに発報できます。
- **RUMと相関分析** します。サイトがRUM計装されていて実ユーザーデータでも同じリグレッションが見られる場合、本番に影響する問題があります。Syntheticだけがリグレッションしている場合、多くの場合サードパーティの依存関係か、ユーザーが現在アクセスしていないコードパスが原因です。
- Scatterplotに戻って **ロケーション間で実行を比較** します。同じジャーニーでメルボルンがロンドンより一貫して8秒遅い場合、APACに対するCDNカバレッジが定量化可能な実際のダメージを与えています。

この実行はテスト履歴の一部として永続的に保存されます。このページのすべてのメトリクスは時系列データでもあり、チャート化、ダッシュボード化、アラート設定が可能です。
