前のステップで表示した Scatterplot で、いずれかのドットをクリックしてテスト実行データにドリルダウンします。可能であれば、サイトの現在の状態にできるだけ近づけるため、最新のテスト実行（最も右側）を選択してください。

![Drilldown](../../img/drilldown.png)

このページ 1 つに、特定のロケーションからの単一の実行について Splunk Synthetic Monitoring が記録した**すべて**の情報が集約されています。各セクションの役割を理解しておく価値があります。リグレッションのトリアージを行う際、最も時間を費やすのがこれらのビューだからです。

## 結果ヘッダー

ページ上部には、実行の概要が表示されます。

- **Status** — Success または Failure。失敗があった場合は、失敗したステップが名前で示されます（前章でステップに名前を付け直した取り組みが、ここで活きてきます）。
- **Location** — 実行元の AWS リージョン。リージョン固有のリグレッションを追跡している場合は、Scatterplot と照らし合わせて確認します。
- **Timestamp** — 実行が開始された時刻（ローカルタイムゾーン）。
- **Total duration** — ジャーニーがエンドツーエンドで完了するまでの所要時間。

## フィルムストリップとビデオ

ページ上部に並ぶスクリーンショットの帯は **filmstrip** と呼ばれるものです。Synthetic Monitoring は実行中、一定間隔でスクリーンショットを取得するため、ジャーニー全体を視覚的にスクラブして確認できます。右側にはテストから見たページの **video replay** があり、標準的なプレーヤーでフレームごとにスクラブ可能です。

filmstrip と video は、「ユーザーが実際に何を見たのか」を最速で把握する手段です。予期しないモーダルの出現、コンテンツシフトによるエラー、要素の欠落などでステップが失敗した場合に特に有用です。バグを再現可能にするために開発者と共有すべき情報は、これらだけで十分なケースが多くあります。

## Business Transactions と Pages

filmstrip の下には、2 つのサマリー行が表示されます。

- **Business Transactions** — ステップエディターで定義した synthetic transactions です（このワークショップでは定義していないため、デフォルトの「Synthetic transaction 1」エントリーが表示されます）。各トランザクションは、それぞれ独自の所要時間を持つ独立したバーとして表示されます。
- **Pages** — ジャーニーが訪問した個別の URL すべてが順番に表示されます。Online Boutique のテストでは、ホームページ、Vintage Camera Lens の商品ページ、カート、チェックアウト確認ページの 4 つが表示されます。いずれかをクリックすると、下部の waterfall がそのページのリソースのみにフィルタリングされます。

## Waterfall

waterfall は結果ページの中核となる機能です。ページが発行したすべての個別の HTTP リクエストが、タイミングとステータスとともに順序通りに表示されます。フィルターバーでリソース種別（**All / XHR / JS / CSS / Img / Media / Font / Doc / WS / Manifest**）による絞り込みが可能で、各行には次の情報が表示されます。

- **METHOD** — 通常は `GET` ですが、API への POST もここに表示されます。
- **FILE** — リソースのパス。行をクリックすると詳細なタイミング情報（DNS、connect、TLS、request、response、content download）が展開されます。
- **DOMAIN** — サードパーティ由来の負荷を見つけるのに有用です。`cdn.googletagmanager.com` や A/B テスト SDK が遅延リストの上位に常に登場している場合、遅延読み込みや削除の候補が見つかったということです。
- **SIZE** — 通信路上のサイズ。意外な値（4 MB のヒーロー画像、ミニファイされていないバンドルなど）に注意します。
- **STATUS** — HTTP ステータスコード。赤いものはすべて調査する価値がありますが、失敗はしていなくてもレイテンシーを増加させる予期しない `301`/`302` のチェーンにも注目してください。

## Web Vitals

ページ右側には、この実行に対する [Core Web Vitals](https://web.dev/articles/vitals) が Splunk によって表示されます。

- **Largest Contentful Paint (LCP)** — 画面内に表示される最大のコンテンツ要素（通常はヒーロー画像やメイン見出し）のレンダリング完了までの時間。Good: ≤ 2.5s、改善が必要: ≤ 4s、Poor: > 4s。
- **Total Blocking Time (TBT)** — ページ読み込み中に長時間タスクによってメインスレッドがブロックされた合計時間。First Input Delay のラボ環境版に相当します。Good: ≤ 200ms、Poor: > 600ms。
- **Cumulative Layout Shift (CLS)** — 読み込み中にレイアウトが予期せずシフトした程度（読み始めた後にバナーが記事を押し下げるような状況を想像してください）。Good: ≤ 0.1、Poor: > 0.25。

これらは Google 標準の Web Vitals メトリクスであるため、他の RUM ツールと同じようにチャート化やアラート設定が可能です。RUM と異なり、synthetic 版は管理された環境から取得されるため、synthetic LCP のリグレッションは、ユーザー環境の変化ではなく、ほぼ確実にコード変更が原因です。

## 次に取り組むこと

このビューに慣れたら、実際の本番運用で役立つフォローアップは次のようなものです。

- ここに表示されるメトリクス（実行時間、ステップ時間、個々の web vitals）に対して **detectors** を作成しましょう。Splunk Synthetic Monitoring の detectors は、他の Observability Cloud detectors と並列で動作し、同じ通知チャネルに発火させることができます。
- **RUM と相関分析する**。サイトに RUM を導入していて、実ユーザーデータでも同じリグレッションが見られる場合、本番に影響する問題が発生しているということです。synthetic だけでリグレッションが見られる場合は、サードパーティの依存関係や、現時点でユーザーが通っていないコードパスが原因であるケースが多くあります。
- Scatterplot に戻って **複数のロケーションでの実行を比較する**。同じジャーニーで Melbourne が London より一貫して 8 秒遅い場合、APAC 向けの CDN カバレッジが定量化可能な実害をもたらしているということです。

この実行は、テストの履歴に永続的に記録されます。このページ上のすべてのメトリクスは時系列データでもあり、チャート化、ダッシュボード化、アラート設定が可能です。
