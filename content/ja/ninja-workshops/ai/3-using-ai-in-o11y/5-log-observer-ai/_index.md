## Log ObserverのAI機能

Splunk Observability CloudのLog Observerには、大量のログデータを理解するためのAI機能がいくつか含まれています。

- **Pattern Detection**: ログ内の共通パターンを自動的に識別
- **Anomaly Highlighting**: 異常なログエントリやパターンを表面化
- **Log Clustering**: 類似したログメッセージをグループ化
- **Intelligent Filtering**: コンテキストに基づくAI提案フィルター
- **Natural Language Queries**: 自然言語によるログクエリ（利用可能な場合）

## パターン検出

### ログパターン検出とは

個々のログ行を表示する代わりに、パターン検出はログを構造ごとにグループ化し、以下を容易にします。

- 頻出パターンとレアパターンの識別
- 新しいまたは異常なログパターンの発見
- 異常なエントリへの集中
- 繰り返しログによるノイズの削減

### 仕組み

AIエンジンは以下を行います。

1. ログメッセージの構造を分析
2. 可変部分（ID、タイムスタンプ、値）を識別
3. パターンテンプレートを作成
4. テンプレートごとにログをグループ化
5. パターンの頻度と変化を追跡

## ハンズオン演習: ログパターンの探索

{{% notice title="演習" style="primary" icon="tasks" %}}

### ステップ1: Log Observerにアクセス

1. **Log Observer** に移動します
2. アクティブなログデータがある時間範囲を選択します
3. 多様なログがあるサービスまたはインデックスを選択します

### ステップ2: パターンビューを有効化

1. **Patterns** または **Pattern Analysis** ビューを探します
2. 個別ログビューからパターンビューに切り替えます
3. ログがどのようにクラスタリングされているか確認します

### ステップ3: パターンを分析

パターンリストを確認します。

- **高頻度パターン**: 正常で予想されるログ
- **新しいパターン**: 最近出現したもの（潜在的な問題）
- **レアパターン**: 低頻度で調査の価値あり
- **エラーパターン**: 構造化されたエラーメッセージ

### ステップ4: 異常を調査

1. レアまたは新しいパターンをクリックします
2. そのパターンのサンプルログを表示します
3. ベースライン/正常パターンと比較します
4. 問題を示しているかどうかを判断します

{{% /notice %}}

## ログ異常検出

### 検出される異常の種類

1. **頻度の異常**: ログ量の急激な増加/減少
2. **コンテンツの異常**: 異常なフィールド値やメッセージ内容
3. **パターンの異常**: ベースラインにない新しいパターン
4. **タイミングの異常**: 通常とは異なる時間に出現するログ

### 異常検出の設定

MLを使用したログベースのディテクターを設定します。

1. **Alerts & Detectors** に移動します
2. Log Observerから新しいディテクターを作成します
3. **Anomaly Detection** モードを選択します
4. 以下を設定します
   - ベースライン期間
   - 感度レベル
   - アラート条件
   - 通知チャネル

## インテリジェントログフィルタリング

### AI提案フィルター

調査中にLog Observerが以下を提案する場合があります。

- **関連フィールド**: フィルタリングすべきタグや属性
- **一般的な値**: 頻出するフィールド値
- **異常な値**: 注意が必要な通常とは異なるフィールド値
- **相関属性**: 一緒に出現することが多いフィールド

### 提案フィルターの使用

1. UI内でフィルター提案を探します
2. クリックして提案フィルターを適用します
3. 結果に基づいて絞り込みます
4. 有用なフィルターの組み合わせを保存します

## ログとAPM・インフラストラクチャの相関

### 自動コンテキストリンク

AIはログを以下と接続するのに役立ちます。

- **Traces**: ログエントリを分散トレースにリンク
- **Services**: ログをAPMサービスに関連付け
- **Infrastructure**: ホスト、コンテナ、Podに接続
- **Metrics**: ログパターンとメトリクスの変化を相関

### AIの手がかりをたどる

調査中の手順です。

1. ログエントリまたはパターンから開始します
2. **Related Content** の提案を探します
3. 相関するトレース、メトリクス、またはインフラストラクチャにジャンプします
4. Tag Spotlightを使用して問題を絞り込みます
5. ログに戻って調査結果を確認します

## ログの要約とインサイト

### 主要インサイトパネル

AI生成のインサイトには以下が含まれる場合があります。

- **エラー率の要約**: エラータイプ別にグループ化
- **サービスの健全性**: ログの重大度に基づく
- **トレンド分析**: 時間経過によるログパターンの変化
- **比較分析**: 現在とベースライン期間の比較

### インサイトの例

```text
⚠️ Error rate increased 340% in the last hour
   → Top error: "Database connection timeout" (1,247 occurrences)
   → Affected services: checkout-service, payment-service
   → Started at: 14:23 UTC

📊 New log pattern detected
   → "WARN: Cache miss for key {key}" appeared 892 times
   → First seen: 14:25 UTC
   → May indicate cache invalidation issue
```

## AIを活用したログ分析のベストプラクティス

### 1. ログを構造化する

構造化ログを使用してAIを支援します。

```json
{
  "timestamp": "2024-01-15T14:23:45Z",
  "level": "ERROR",
  "service": "checkout-service",
  "message": "Payment processing failed",
  "error_code": "PAYMENT_TIMEOUT",
  "transaction_id": "txn_123456",
  "customer_tier": "enterprise"
}
```

### 2. 一貫したフィールド名を使用する

- サービス間でフィールド命名を標準化します
- 共通の分類法を使用します（例: `service.name`、`serviceName` と `service_id` の混在を避ける）
- すべてのログに必要なコンテキストを含めます

### 3. 適切なログレベルを設定する

- **DEBUG**: 詳細な診断情報（開発用）
- **INFO**: 一般的な情報メッセージ
- **WARN**: 潜在的に有害な状況
- **ERROR**: アプリケーションの継続は可能だがエラーが発生したイベント
- **FATAL**: アプリケーションの早期終了を引き起こす重大なエラー

### 4. ログサンプリングを活用する

大量のログの場合は以下を行います。

- AIを使用して代表的なサンプルを特定します
- エラーログと異常に注力します
- インテリジェントサンプリングを適用してコストを削減します

### 5. ログベースのディテクターを作成する

以下に対するアラートを設定します。

- 重大なエラーパターン
- 異常なログ量
- 新しいエラータイプ
- セキュリティ関連のパターン

## ユースケース

### ユースケース1: メモリリークの特定

**観察結果**: パターン分析で「GC pressure」の警告が増加

**AIによる支援内容**:

- GC関連のログパターンをグループ化
- 頻度の増加をハイライト
- メモリメトリクスとの相関
- 影響を受けるサービスのトレースにリンク

### ユースケース2: セキュリティ問題の検出

**観察結果**: 新しいパターン「Authentication failed」が出現

**AIによる支援内容**:

- 新規/レアパターンとしてフラグ付け
- ソースIP別にクラスタリング
- 異常なアクセスパターンをハイライト
- セキュリティ関連のフィルターを提案

### ユースケース3: データベースパフォーマンスの劣化

**観察結果**: スロークエリの警告が増加

**AIによる支援内容**:

- クエリをパターン別にグループ化
- 最も遅いクエリタイプを特定
- データベースメトリクスとの相関
- アプリケーショントレースにリンク

## 制限事項と考慮事項

- **パターンの品質はログの構造に依存**: 非構造化ログはパターン化が困難
- **高カーディナリティフィールド**: UUIDやユニークIDがパターンを分割する可能性
- **学習期間**: AIは異常検出のためにベースラインデータが必要
- **コンテキストが重要**: ログのAIを他のオブザーバビリティシグナルと組み合わせる

{{% notice title="ヒント" style="info" icon="lightbulb" %}}
最も効果的なログ分析は、AIによるパターン検出とドメイン知識を組み合わせたものです。AIを使用してシグナルを表面化させ、専門知識を適用して解釈します。
