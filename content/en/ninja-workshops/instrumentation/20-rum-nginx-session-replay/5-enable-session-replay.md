After RUM data is visible, configure DXA to treat the RUM application as a product analytics workspace.

## Create a DXA project

In Splunk Observability Cloud:

1. Open **Digital Experience Analytics**.
2. Select **Create project**.
3. Name the project `DXA RUM Workshop`.
4. Add the RUM application from `.env`, for example `dxa-rum-workshop`.
5. Add a description such as `Checkout journey analysis for the RUM workshop`.
6. Create the project.

Projects group applications, shared event definitions, analyses, and team access. A production project might map to a product, business unit, campaign, or cross-application journey.

## Create event definitions

Open the project and select **Event Definitions**. Create these definitions from the RUM events generated by the demo app:

| Event definition | Source event pattern | Why it matters |
| --- | --- | --- |
| `Product viewed` | Navigation or route change where URL contains `/products/` | Entry point for the product journey. |
| `Add to cart` | Click on text or element matching `Add to cart` | First conversion signal. |
| `Begin checkout` | Click on text or element matching `Begin checkout` | Checkout funnel start. |
| `Order submitted` | Submit or click event on the checkout form button | Final conversion signal. |
| `Checkout error` | JavaScript error from the demo page | Technical friction that can explain drop-off. |

Use **New event definition** for each event. Build the definition from predefined RUM events, saved definitions, or custom event data.

## Optional: use the element picker

If the DXA element picker can reach the demo page:

1. From **Event Definitions**, select **Pick from page**.
2. Enter the demo URL.
3. Turn on the element picker.
4. Select the button or form control you want to track.
5. Use the selection slider if the captured selector is too narrow or too broad.
6. Choose whether to track the element on all URLs or only the current URL.
7. Confirm the event definition.

{{% notice title="Selector hygiene" style="warning" %}}
Avoid brittle selectors based only on generated DOM position. Prefer stable IDs, semantic button text, or durable data attributes that the frontend team agrees to keep.
{{% /notice %}}

Which events should become DXA event definitions?
{{% /tab %}}
{{% tab title="Answer" %}}
Start with behavior that maps to a product question: funnel steps, adoption of a released feature, high-value interactions, abandonment points, and error states. Do not turn every browser event into a saved definition.
