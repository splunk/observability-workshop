# Shortcode Reference

The `hugo-theme-splunk-workshop` theme provides shortcodes (Hugo's mechanism for parameterized template fragments inside markdown). The repo uses a small, consistent set of them. This file is the catalog.

A note on syntax: Hugo distinguishes `{{% ... %}}` (markdown content inside) from `{{< ... >}}` (raw HTML inside). Most workshop notices use `{{% ... %}}` because the body contains markdown lists. Tabs use `{{< tabs >}}` because the wrapper is structural.

## Notice — the workhorse

The notice block visually chunks page content into recognizable boxes. It's by far the most common shortcode in workshops.

```markdown
{{% notice title="Exercise" style="green" icon="running" %}}

* Step 1...
* Step 2...

{{% /notice %}}
```

### Standard variants

| Purpose | Style | Icon | Title |
|---------|-------|------|-------|
| Hands-on exercise (the user clicks/types) | `green` | `running` | `"Exercise"` |
| Persona / scene-setting at start of chapter | `orange` | `user` | `"Persona"` |
| Sidebar context, "what is this feature?" | `info` | (none) | feature name |
| Final celebration | `blue` | `wine-bottle` | `"Congratulations"` |

The icons come from FontAwesome/Lucide integrations exposed by the theme. Icons are optional - omit `icon=` when unnecessary.

### When to use each

- **Exercise** is for "do this" instructions. Almost every topic page has at least one.
- **Persona** sets up the chapter's framing — it says who the reader is in this exercise (frontend engineer, SRE, back-end dev). It's almost always at the top of a chapter `_index.md`. The persona block establishes context for the chapter; subsequent topic pages don't repeat it.
- **Info** is for explanatory asides — "Trace Analyzer is a tool that..." — that the reader can skim or skip. Don't put step-by-step instructions in an info notice; that belongs in an Exercise.
- **Congratulations** appears at the end of a chapter when there's a meaningful milestone, and once in the wrap-up chapter.

## Splunk callouts (`> [!splunk]`)

This is a custom admonition the theme renders with a Splunk-branded panel. It looks like a GitHub-style callout but with Splunk styling.

```markdown
> [!splunk] We are going to look at the *Real User Monitoring* (**RUM**) data generated by you and your fellow attendees while using the Online Boutique.
```

Use this for **scene-setting copy** — explaining what the upcoming exercise demonstrates and why it matters. Typically appears inside a Persona notice in chapter `_index.md` files.

## Tabs — Question/Answer reveals

A click-to-reveal pair. Used so the user can pause and think before reading the answer.

```markdown
{{< tabs >}}
{{% tab title="Question" %}}
**What can you conclude from the chart in the side pane?**
{{% /tab %}}
{{% tab title="Answer" %}}
**The error percentage is very high.**
{{% /tab %}}
{{< /tabs >}}
```

Conventions:

- Phrase the question as a **bold sentence ending in a question mark**.
- Phrase the answer as a **bold short statement**, sometimes with a single supporting detail.
- For multi-part questions, use `title="Questions"` / `title="Answers"` (plural) and number both lists 1, 2, 3.
- Tabs blocks appear *inside* an Exercise notice when the question is part of the click-through flow, or as a standalone block when the question is reflective.

## Buttons — UI element references

When the instructions tell the user to click an actual button in the Splunk Observability UI:

```markdown
* Click the {{% button style="blue" %}}Apply{{% /button %}} button.
* Click the {{% button style="grey" %}}Breakdown{{% /button %}} dropdown.
* Click the {{% button style="red" %}}!{{% /button %}} marker on the span.
```

Style mapping (matches Observability Cloud's actual button colors):

- `blue` — primary action buttons (Apply, Run Search, Save).
- `grey` — secondary buttons and dropdowns (Breakdown, Cancel).
- `red` — destructive or alert markers (the red `!` on error spans).
- `green` (rare) — success/confirm buttons.

You can omit `style=` and the theme uses a default grey:

```markdown
* Click {{% button %}}**Create Detector**{{% /button %}}.
```

## Icon references (`{{% icon %}}`)

Inline FontAwesome icons:

```markdown
* Click on the blue {{% icon icon="link" %}} **APM** link...
```

Use sparingly — only when the visual cue is meaningful (a chain link icon next to a link, a warning triangle next to an alert).

## Badge — for inline labels

```markdown
{{< badge style="blue" icon="fab fa-fw fa-linkedin-in" >}}{{< linkedin >}}{{< /badge >}}
```

Mostly used in the wrap-up chapter to render a "share to LinkedIn" badge. Not commonly needed in regular workshop content.

## Image syntax

Standard markdown, with an optional Hugo `?width=` query string:

```markdown
![alt text](../images/foo.png)
![alt text](../images/foo.png?width=750px)
![alt text](images/foo.png?width=20vw)
```

- Use `../images/...` from a topic page (image lives at chapter level).
- Use `images/...` from a chapter `_index.md` (image lives in the chapter's own folder).
- The `?width=` is optional — used to constrain wide screenshots that would otherwise dominate the page. Common values: `750px`, `500px`, `20vw`.
- Alt text should describe the screenshot, not just repeat the filename.

## What NOT to use

The theme supports many shortcodes; resist using ones that aren't already common in the repo. Specifically, avoid:

- `{{< highlight >}}` — use markdown fenced code blocks instead.
- `{{% expand %}}` — the tabs Question/Answer pattern is the established way to do reveals.
- Custom HTML — almost everything you'd want a `<div>` for can be expressed with notices.

Consistency matters more than cleverness. A reader skimming workshops should recognize the visual rhythm immediately.
