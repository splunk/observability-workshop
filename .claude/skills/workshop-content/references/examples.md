Notes:

- The Exercise notice is the wrapper for the whole click-through.
- A nested `{{< tabs >}}` block sits *inside* the Exercise — the question is part of the flow.
- An Info notice (`style="info"`) provides context about the *concept* of span tags. It's a teaching moment, not a click instruction.
- After the info notice, the exercise resumes with `* Next, we need to...`. Numbered callouts **(1)** and **(2)** point to spots in the screenshot.
- Bold for UI labels (**paymentservice**, **Traces**), backticks for technical identifiers (`v350.10`, `version`).

title: 6. APM Waterfall
weight: 6
---

The **Trace Waterfall** view displays all spans within a trace as a hierarchical timeline. Each span appears as a horizontal bar, with the bar's length representing its duration and its position showing when it occurred relative to other spans.

A trace is a collection of spans that share the same trace ID, representing a unique transaction handled by your application and its constituent services.

A span represents a single unit of work within a trace, capturing information about a specific operation such as an API call, database query, or service request. Each span includes metadata like the operation name, start time, duration, and associated tags or attributes that provide context about the work being performed.

## A note on `[NAME OF WORKSHOP]`

The short workshop intentionally uses the placeholder `[NAME OF WORKSHOP]` in instructions like:

```markdown
* The **Environment** selected is **[NAME OF WORKSHOP]-workshop**.
```

This is so the same workshop can be delivered at different events ("Cisco Live", "SplunkLive London") — the instructor does a find-and-replace before the session. If you're writing a workshop tied to a specific event, replace it with the actual name. If it's a generic template, leave it.
